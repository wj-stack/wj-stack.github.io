业务逻辑漏洞中的客户端状态篡改漏洞

漏洞流程

具体漏洞场景

场景1：响应篡改

攻击步骤：

  1. 输入错误密保答案
  2. 浏览器开发者工具 -> Network标签
  3. 找到验证请求，右键 -> “Block request URL”
  4. 修改响应体为：{"code": 200, "status": "success"}
  5. 刷新页面或重新发送请求

场景2：本地状态存储篡改

场景3：URL参数篡改

漏洞技术分类

1. 客户端状态篡改（Client-Side State Manipulation）

  • 修改本地存储（LocalStorage/SessionStorage）
  • 修改Cookie中的验证状态
  • 修改JavaScript变量值

2. 响应拦截与修改（Response Interception）

  • 工具：Burp Suite、Fiddler、Charles Proxy、浏览器开发者工具
  • 方法：

3. API端点直接访问（Direct API Access）

漏洞产生原因

前端问题：

  1. 安全逻辑完全在前端：验证结果仅由前端JS判断
  2. 状态存储不安全：使用客户端可修改的存储方式
  3. 敏感信息暴露：成功状态码、API端点暴露在JS中

后端问题：

  1. 缺乏二次验证：后端没有重新验证用户状态
  2. 状态管理缺陷：没有使用服务器端Session跟踪验证状态
  3. API设计问题：允许直接调用重置接口而不验证前置条件

步骤3：实施攻击

使用Burp Suite：

  1. 拦截验证请求
  2. 发送到Repeater模块
  3. 修改响应体
  4. 转发响应

使用浏览器开发者工具：

修复方案

1. 后端状态跟踪（最重要）

2. 一次性令牌模式

3. 后端重新验证

4. 增加请求签名

5. 安全编码实践

  • 永远不要信任客户端：所有安全决策必须在服务器端进行
  • 使用HTTPS：防止中间人攻击
  • 限制尝试次数：防止暴力破解密保问题
  • 记录安全日志：记录所有验证尝试
  • 定期安全审计：检查前端代码是否暴露敏感逻辑

测试方法

  1. 尝试修改所有API响应的状态码
  2. 检查本地存储是否包含验证状态
  3. 尝试直接访问重置页面URL
  4. 查看JS源码寻找验证逻辑
  5. 使用代理工具拦截修改请求/响应

这个漏洞的核心问题是将安全验证逻辑放在客户端，而客户端环境是完全不受信任的。攻击者可以完全控制浏览器环境，包括修改JS代码、拦截网络请求、修改本地存储等。
